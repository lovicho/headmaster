# ✅ 해마(Headmaster) 검증 기준 (VERIFY)

> **PVE의 VERIFY 단계 산출물** — 구현 전에 검증 기준을 먼저 정의한다.
> 실행 명령은 모두 `backend/` 디렉터리 기준.

---

## 0. 공통 게이트 (모든 Phase, 모든 커밋)

```powershell
uv run ruff check .          # 린트 통과
uv run mypy headmaster       # strict 모드 타입 체크 통과
uv run pytest                # 전체 테스트 통과
```

- Python 코드: type hints 필수, `Any` 남용 금지 (TS strict 규칙의 Python 등가)
- 하드코딩된 시크릿/API 키 금지 — 모델 키는 환경변수만
- 커밋: Conventional Commits

## 2. Phase 1 게이트 — 코어 MVP

| # | 기준 | 검증 방법 |
|---|------|----------|
| 1-1 | `headmaster run "<작업>"` 1회로 TaskSpec→하네스→실행→Critic→publish E2E 동작 | 수동 + E2E 테스트 |
| 1-2 | I-B-F Proof 누락 산출물이 Critic에서 자동 REJECTED (`zero_shot_detected: true`) | `test_critic_service.py` |
| 1-3 | 동일 이벤트 로그에서 상태 replay 100% 일치 | `test_event_replay.py` |
| 1-4 | 어댑터 2종(Anthropic/OpenAI)이 동일 계약 테스트 스위트 통과 — LLM-agnostic 입증 | `test_model_gateway.py` (mock + 선택적 live) |
| 1-5 | 모델/도구 호출이 전부 이벤트로 기록됨 (누락 0) | replay 테스트로 간접 검증 |
| 1-6 | LLM 호출 없는 단위 테스트는 전부 오프라인 실행 가능 | CI에서 키 없이 pytest 통과 |

## 3. Phase 2 게이트 — 도구 + 지식 순환

- 작업 2건 연속 실행 시 2번째가 1번째 자산을 Imitate 베이스로 재사용 (trace로 확인)
- 검증 실패 기록이 quarantine에 격리되고 일반 검색에서 제외됨
- allowlist 밖 MCP 도구 호출이 policy_engine에서 거부됨

## 4. Phase 3 게이트 — 보증 평면

- 고위험 작업(`needs_human_approval: true`)이 승인 없이 publish되지 않음
- budget soft limit → 모델 다운그레이드 / hard limit → 중단 + 승인 요구
- golden set 회귀 시 배포 차단 (Eval Runner)
- 감사 로그에 who decided / based on what / who approved 기록

## 4.5 Phase 4a 게이트 — 제어 API

- `POST /v1/tasks` 제출 → 백그라운드 실행 → `GET /v1/tasks/{id}`로 상태(이벤트 replay 기반) 조회
- 고위험 작업이 `GET /v1/approvals` 큐에 잡히고, HTTP 승인/거부가 실행을 재개/차단함
- 미존재 리소스 404, `/v1/metrics`·`/v1/evals/run` 정상 동작
- 검증: `test_api.py` (ASGI 오프라인) + `headmaster serve` 라이브 스모크
