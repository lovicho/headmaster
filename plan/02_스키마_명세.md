# 📐 해마(Headmaster) 스키마 명세 v1.0

> **Phase 0 산출물** — 코드보다 계약 먼저 (1st 보고서 "정책 우선" 원칙)
> **정본(canonical)**: `backend/headmaster/schemas/`의 Pydantic v2 모델. 본 문서는 설계 의도와 필드 의미를 기록한다.
> **표기 원칙**: English-Core — 모든 스키마 필드명·열거값은 EN. 본 문서의 설명만 KO.

---

## 1. 6대 스키마 개요

| # | 스키마 | 역할 | 근거 |
|---|--------|------|------|
| 1 | `TaskSpec` | 사용자 요청의 정규화된 기계 표현 | 2nd·3rd 보고서 통합 |
| 2 | `AgentManifest` | 에이전트의 능력·비용·권한 선언 | 2nd 보고서 |
| 3 | `HarnessManifest` (Agent/Orchestra) | 정책화된 실행 패키지 계약 | 1st 보고서 manifest + 테스트설계 v8 |
| 4 | `EvidenceBundle` (+ `IBFProof`) | 주장-증거 객체화, I-B-F 증빙 | 2nd 보고서 + v8 IBF_Proof |
| 5 | `CritiqueReport` | Critic 게이트의 판정 결과 | 3rd 보고서 + v8 Critic 출력 스키마 |
| 6 | `MemoryRecord` | 5계층 메모리의 단일 레코드 | 2nd 보고서 |

보조: `TaskState`(상태머신), `Event`(CloudEvents 정렬 이벤트 봉투).

## 2. TaskSpec

| 필드 | 타입 | 설명 |
|------|------|------|
| `task_id` | str (`tsk_` 접두) | 고유 ID. MVP는 uuid4 hex, 추후 UUIDv7 전환(3rd 권장) |
| `title`, `intent` | str | 제목 / 목표 서술 |
| `constraints.language` | `ko`\|`en` | 대면 출력 언어 (기본 ko — Korean-Edge) |
| `constraints.deadline` | datetime? | 기한 |
| `constraints.budget` | Budget | `max_model_cost_usd`, `max_tool_calls`, `max_tokens` |
| `constraints.quality_criteria` | list[str] | 품질 기준 |
| `risk_profile.data_sensitivity` | `public`\|`internal`\|`confidential` | 데이터 민감도 |
| `risk_profile.action_risk` | `low`\|`medium`\|`high` | 행위 위험 |
| `risk_profile.needs_human_approval` | bool | HITL 게이트 필요 여부 |
| `inputs[]` | {type, ref, trust} | 입력 자료 (file/url/text, 신뢰 출처 표기) |
| `success_criteria[]` | list[str] | 성공 판정 기준 |
| `required_evidence[]` | list[str] | 요구 증거 수준 |
| `policy_checks[]` | list[str] | 적용할 정책 체크 ID |

## 4. HarnessManifest

**하네스 = "프롬프트 묶음"이 아니라 정책화된 실행 계약** (1st 보고서). 두 종류:

### 4.1 AgentHarness (`kind: agent`) — v8 개별 에이전트 하네스의 기계 표현

| 필드 | 설명 |
|------|------|
| `harness_id`, `version` | 식별자, semver (v8 이식분은 `8.0.0`) |
| `role` | 에이전트 역할 (AgentManifest.role과 매칭) |
| `persona.role`, `persona.objective` | v8 "System Persona & Role" 절 |
| `inherited_directives[]` | v8 "Inherited Master Directives" 절 |
| `ibf_protocol.imitate/benchmark/fusion/maintain` | v8 "Specific I-B-F Protocol" 절 (역할에 따라 일부 생략 가능, 최소 1개 필수) |
| `output_contract.format` | `json`\|`markdown`\|`json+markdown` |
| `output_contract.schema_ref` | 출력 스키마 참조 (있을 시 기계 검증 대상) |
| `language_policy.internal/external` | EN-Core / KO-Edge |
| `tool_policy` | MCP allowlist, sandbox 필수 여부, 외부 쓰기 승인 |
| `cost_tier` | 모델 티어 |

### 4.2 OrchestraHarness (`kind: orchestra`) — Master Orchestrator v8 + 1st manifest의 기계 표현

| 필드 | 설명 |
|------|------|
| `harness_id`, `version`, `task_class`, `risk_tier` | 1st 보고서 manifest 필드 |
| `core_directives[]` | v8 Core Directives (No Zero-Shot 등) |
| `required_roles[]` | 필요 에이전트 역할 목록 |
| `phases[]` | Phase 0~7: `{phase_id, title, agents, description, gate}` |
| `phases[].gate` | `{gate_id, description, approver}` — approver: `critic`\|`human`\|`secops_qa` |
| `ibf_requirements` | `must_reference_internal_assets`, `must_reference_external_benchmarks`, `proof_format[]` |
| `approval_gates[]`, `tool_policy`, `eval_suite[]`, `rollback_policy` | 1st manifest 동일 |

### 4.3 v8 → YAML 변환 규칙

1. v8 마크다운의 4개 절(`Persona & Role` / `Inherited Master Directives` / `Specific I-B-F Protocol` / `Output Schema`)을 위 필드로 1:1 매핑
2. 산문 지시는 `inherited_directives` 리스트 항목으로 분해 (1항목 = 1지시)
3. Output Schema의 JSON 예시는 `output_contract.schema_ref`로 스키마명만 참조 — 실제 검증은 Pydantic 모델이 담당
4. 언어 규칙: 내부 전용 에이전트(`knowledge_manager`, `researcher`, `critic`, `design`, `dev_fe_be`, `secops_qa`)는 `external: en`, 대면 출력 에이전트(`consultant`, `planner`, `content`)는 `external: ko`
5. SecOps와 QA는 v8 원문대로 단일 하네스(`secops_qa`)로 통합 유지

## 5. EvidenceBundle + IBFProof

**No Zero-Shot Invention의 기계적 강제 장치.** 텍스트 설명이 아니라 4개 필드의 존재·일관성으로 검증 (1st 보고서 "provenance-aware orchestration").

```
EvidenceBundle
├── bundle_id (evb_), task_id
├── ibf_proof: IBFProof
│   ├── imitated_assets[]        # 내부 RAG 자산 ID (Imitate 증빙)
│   ├── benchmarked_references[] # 외부 레퍼런스 URI (Benchmark 증빙)
│   ├── fusion_method            # 융합 방법 서술
│   ├── trace_ids[]              # 실행 trace span ID
│   └── artifact_hashes[]        # 산출물 해시
├── claims[]: Claim
│   ├── claim_id, text
│   ├── claim_type: factual_assertion | design_inference
│   ├── supports[]: {source, kind: file|web|rag_asset|trace|artifact}
│   └── confidence: float [0,1]
├── counterevidence[]: Claim
└── verifier_status: pending | pass | fail | revise
```

## 6. CritiqueReport

v8 Critic 출력 스키마와 **하위 호환** (v8 예시 JSON이 그대로 검증을 통과해야 함 — Phase 0 게이트), 3rd 보고서 findings 구조를 확장으로 수용.

| 필드 | 타입 | 비고 |
|------|------|------|
| `target_agent` | str | v8 필수 |
| `status` | `APPROVED`\|`REJECTED` | v8 필수 |
| `zero_shot_detected` | bool | v8 필수 |
| `verification_details` | {imitation_check, benchmark_check, fusion_coherence} | v8 필수 |
| `mandatory_revisions[]` | list[str] | v8 |
| `findings[]` | {issue_type, severity, description, proposed_fix} | 3rd 확장. issue_type: `missing_evidence`\|`logic_gap`\|`policy_violation`\|`format_error`\|`zero_shot_invention` |
| `requires_human_approval`, `requires_replan` | bool | 3rd 확장 (기본 false) |
| `task_id`, `critique_id` | str? / str | 확장 (v8 예시엔 없으므로 optional/자동생성) |

## 7. MemoryRecord

| 필드 | 타입 | 설명 |
|------|------|------|
| `memory_id` | str (`mem_`) | |
| `scope` | `stm`\|`episodic`\|`semantic`\|`skill`\|`evidence` | 5계층 (2nd) |
| `task_id` | str? | 출처 작업 |
| `timestamp` | datetime (UTC) | |
| `summary` | str | |
| `embedding_ref` | str? | 벡터 참조 (Phase 2) |
| `salience` | float [0,1] | 작업 영향도 — 승격 게이트 입력 |
| `confidence` | float [0,1] | 증거 신뢰도 — 승격 게이트 입력 |
| `decay_policy` | `reinforce_on_reuse`\|`fixed_ttl`\|`immutable` | EVID는 immutable |
| `source_refs[]` | list[str] | |
| `quarantine` | bool | **검증 실패 기록 격리 (폐기 금지)** — 기본 false |
| `reuse_count` | int | 승격 조건 (≥2) 추적용 |

승격 게이트(Phase 2 구현 예정): `salience ≥ 0.75 AND confidence ≥ 0.80`에서 consolidation 후보, `reuse_count ≥ 2 AND confidence ≥ 0.85`에서 SEM 승격, SKILL은 `test_pass_rate ≥ 0.9` 추가.

## 8. 상태머신 (TaskState)

3rd 보고서 상태머신 채택. Critiquing/Replanning은 1급 상태 (내부 생각이 아니라 추적 가능한 상태).

```
REGISTERED → CLASSIFIED → PLANNED → EXECUTING
EXECUTING ⇄ AWAITING_TOOL
EXECUTING ⇄ AWAITING_HUMAN_APPROVAL
EXECUTING → CRITIQUING → VALIDATED → PUBLISHING → ASSIMILATING → COMPLETED
CRITIQUING → REPLANNING → PLANNED          (반려 → 재계획)
CRITIQUING → AWAITING_HUMAN_APPROVAL       (고위험 판정 / human 게이트)
CRITIQUING → PLANNED                       (오케스트라: 페이즈 게이트 통과 → 다음 페이즈)
AWAITING_HUMAN_APPROVAL → REPLANNING       (승인 거부 → 재계획)
AWAITING_HUMAN_APPROVAL → PLANNED          (페이즈 게이트 승인 → 다음 페이즈)
AWAITING_HUMAN_APPROVAL → VALIDATED        (발행 승인)
{EXECUTING, AWAITING_TOOL} → RECOVERING → EXECUTING   (장애 복구)
임의 비종료 상태 → FAILED
종료 상태: COMPLETED, FAILED
```

전이 규칙은 `schemas/states.py`의 `ALLOWED_TRANSITIONS`가 정본. 위반 시 `InvalidTransitionError`.

## 9. 이벤트 (CloudEvents 정렬)

**이벤트 로그가 상태의 source of truth** (event sourcing). replay = 기록된 이벤트의 재생 (LLM 재호출 아님 → 결정성 보장).

봉투: `{specversion: "1.0", id: evt_*, source, type, time, subject(=task_id), data}`

| 이벤트 타입 | 발생 시점 |
|------------|----------|
| `task.registered` / `task.classified` | 접수 / 분류·하네스 선택 |
| `harness.compiled` | 하네스 컴파일 완료 |
| `plan.created` | 작업 그래프 생성 |
| `agent.dispatched` | 에이전트 배치 |
| `model.called` / `model.responded` | LLM 호출/응답 (응답 본문 포함 — replay 근거) |
| `tool.called` / `tool.responded` | 도구 호출/응답 |
| `artifact.produced` | 산출물 생성 |
| `critique.issued` | CritiqueReport 발행 |
| `approval.requested` / `approval.granted` / `approval.denied` | HITL 게이트 |
| `replan.triggered` | 재계획 |
| `budget.exceeded` | 예산 초과 (soft/hard) |
| `state.changed` | 상태 전이 (from/to 기록) |
| `artifact.published` | 발행 |
| `knowledge.supplied` | KM의 Imitate 베이스 공급 (bootstrap 여부 포함) |
| `knowledge.assimilated` | 지식 자본화 (v8 Phase 7) — 격리/재사용/승격 기록 포함 |
| `policy.denied` | 정책 거부 (allowlist 밖 도구 호출 등 — 감사 대상) |
| `recovery.started` | 복구 개시 |
| `task.completed` / `task.failed` | 종료 |
